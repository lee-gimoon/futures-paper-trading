모의 선물거래 백엔드 도메인 규칙

이 문서는 현재 프로젝트의 백엔드 코드와 MVP 가정을 기준으로, 그 안에 들어 있는 업무 규칙을 도메인 관점에서 다시 정리한 문서다. 목적은 “지금 코드가 어떤 규칙을 구현하고 있는가”를 한눈에 보는 것이다.

프로젝트가 만드는 것

이 백엔드는 한 사용자가 가입하고, 실시간 BTCUSDT 호가를 기준으로 모의 선물 주문을 넣고, 주문이 체결되면 포지션/손익/마진/청산 상태를 계산해 계좌 화면에 보여주는 시스템이다.

실제 거래소 전체를 그대로 복제하는 문서가 아니라, 현재 MVP가 채택한 선물 모의거래 규칙을 명시한다.

도메인 지도

도메인 핵심 질문
사용자/인증 누가 계좌와 주문의 주인인가? 비밀번호와 사용자 데이터는 어떻게 보호하는가?
시장 데이터 거래소 호가를 우리 서비스의 가격 기준으로 어떻게 해석하는가?
주문 사용자가 낸 주문이 유효한가? 시장가/지정가는 어떤 상태 흐름을 가지는가?
체결 현재 호가를 기준으로 주문을 체결해 보면 몇 개, 어떤 가격에 체결되는가?
포지션/PnL 체결 기록으로 현재 롱/숏 수량, 평균 진입가, 실현/미실현 손익을 어떻게 계산하는가?
레버리지/마진/청산 포지션에 잠긴 담보금(증거금)은 얼마이며, 어느 가격에서 청산되는가?
계좌 평가 프론트 계좌 화면에 필요한 현금, 평가자산, 가용잔고, 포지션 표시값을 어떻게 조립하는가?
이 문서는 현재 구현을 도메인 언어로 풀어 쓴 기준 설명서다.

기준 기록

기준 기록 역할
사용자 계좌, 주문, 체결을 소유하는 주체. 모든 개인 데이터는 사용자 단위로 격리한다.
계좌 사용자의 모의 현금과 신규 주문용 계좌 레버리지를 저장한다.
주문 사용자가 낸 의도와 그 처리 상태를 남기는 기록이다.
체결 주문이 실제로 호가에 닿아 거래된 결과다. 포지션과 실현손익 계산의 원장이다.
호가 스냅샷 한 시점의 매수/매도 호가 목록이다. 주문 체결, 미실현손익, 청산 판정의 시장 가격 기준이다.

저장값과 계산값

이 프로젝트는 모든 값을 저장하지 않는다. 변하지 않는 기준 기록은 저장하고, 체결 원장에서 다시 계산할 수 있는 값은 계산값으로 둔다.

저장한다 계산한다
사용자, 계좌 현금, 계좌 레버리지, 주문, 주문 시점 레버리지, 체결 현재 포지션, 평균 진입가, 실현손익, 미실현손익, 사용 증거금, 청산가, 가용잔고
영속 데이터는 기준 기록이고, 계좌 화면 값은 기준 기록과 현재 시장가격에서 파생된다.

사용자/인증 도메인

규칙 1. 이메일은 사용자 식별자다.

회원가입 시 같은 이메일이 이미 존재하면 새 사용자를 만들 수 없다. 로그인과 사용자 조회의 기준도 이메일이다.

규칙 2. 비밀번호 원문은 저장하지 않는다.

비밀번호는 단방향 해시로 바꿔 저장한다. 로그인 때는 입력한 비밀번호와 저장된 해시를 비교할 뿐, 해시에서 원문을 복구하지 않는다.

규칙 3. 외부 응답에는 민감 정보를 포함하지 않는다.

사용자 응답에는 id, email, displayName 같은 표시용 값만 담는다. 비밀번호 해시는 저장에는 필요하지만 API 응답으로 나가면 안 된다.

사용자 격리 규칙

계좌, 주문, 체결은 모두 특정 사용자에게 속한다. 주문 목록 조회, 주문 취소, 계좌 평가에서 다른 사용자의 데이터를 섞어 보여주거나 수정하면 안 된다.

상황 규칙
주문 생성 현재 로그인 사용자의 주문으로 기록한다.
주문 목록 현재 사용자에게 속한 주문만 반환한다.
주문 취소 주문이 존재해도 소유자가 다르면 취소할 수 없다.
계좌 평가 현재 사용자의 계좌, 주문, 체결만 합산한다.
인증 도메인의 핵심은 “누구의 데이터인가”를 모든 거래 규칙 앞에 세우는 것이다.

시장 데이터 도메인

외부 거래소의 원시 메시지는 그대로 도메인에서 쓰지 않는다. 거래소가 보내는 필드명과 배열 구조를 우리 서비스의 호가 스냅샷 언어로 번역한 뒤 사용한다.

용어 정의
bid 매수 호가. 누군가 이 가격에 사겠다고 걸어 둔 가격과 수량이다.
ask 매도 호가. 누군가 이 가격에 팔겠다고 걸어 둔 가격과 수량이다.
bestBid 매수 호가 중 가장 높은 가격. 지금 시장가로 팔 때 먼저 닿는 가격이다.
bestAsk 매도 호가 중 가장 낮은 가격. 지금 시장가로 살 때 먼저 닿는 가격이다.
mid bestBid와 bestAsk의 중간값. 계좌 평가와 청산 판정에서 현재 가격의 MVP 기준으로 쓴다.
mid = (bestBid + bestAsk) / 2
규칙 1. 가격과 수량은 십진수 정밀도로 다룬다.

거래소 가격과 수량은 손익 계산에 직접 연결되므로 이진 부동소수점 오차가 생기면 안 된다. 문자열 가격은 십진수 숫자로 변환해 보관한다.

규칙 2. 최신 스냅샷은 한 시점의 기준 가격이다.

주문 체결 계산은 계산을 시작한 그 시점의 호가 스냅샷을 기준으로 끝까지 진행한다. 계산 중 새 호가가 도착하더라도 이미 잡은 기준 스냅샷을 중간에 바꾸지 않는다.

실제 거래소는 사용자가 본 호가 스냅샷을 고정해 체결하지 않는다. 주문이 매칭 엔진에 도착하면 주문/취소 이벤트 큐에서 순서가 정해진다.

이벤트 큐 예시: 주문 A → 취소 B → 주문 C

엔진은 이 순서를 지켜 자기 차례가 된 이벤트를 현재 order book에 대해 원자적으로 처리한다. 주문 A 처리 중에는 취소 B나 주문 C가 같은 order book을 중간에 수정하지 못한다. 주문 A 처리가 끝난 뒤 취소 B, 그 다음 주문 C가 처리된다.

즉, 이벤트 하나를 처리하는 동안 같은 order book은 다른 이벤트가 동시에 수정하지 못하므로, 한 주문은 자기 처리 시점의 일관된 주문장 상태를 기준으로 체결된다. 하지만 이 프로젝트는 실제 매칭 엔진 큐 대신 최신 스냅샷 하나를 잡아 같은 일관성을 단순화해서 지킨다.

규칙 3. 호가가 없으면 가격 기반 계산을 할 수 없다.

시장가 체결, 미실현손익, 청산 판정은 현재 가격이 필요하다. 아직 호가가 없거나 한쪽 호가가 비어 있으면 해당 계산은 보류하거나 0/null로 표현한다.

시장 데이터 도메인은 외부 거래소 메시지를 내부 가격 기준으로 번역하는 경계다.

주문 도메인

분류 값 규칙
심볼 BTCUSDT MVP는 단일 심볼만 지원한다. 다른 심볼은 주문 입력에서 거부한다.
방향 BUY / SELL BUY는 매수, SELL은 매도다. 포지션 계산에서는 BUY를 +수량, SELL을 -수량으로 해석한다.
종류 MARKET / LIMIT 시장가는 즉시 체결을 시도하고, 지정가는 가격이 닿지 않으면 대기한다.
수량 양수 0 이하 수량은 주문할 수 없다.
지정가 LIMIT일 때 양수 시장가는 지정가가 없어도 된다. 지정가는 반드시 양수 limit price가 필요하다.

주문 상태

상태 의미
NEW 체결 계산에 넘기는 주문 입력 객체(probe)의 자리 표시 상태다. 현재 구현에서는 DB에 저장되는 업무 상태가 아니라, tryFill 결과로 FILLED/OPEN/REJECTED를 정하기 전의 계산용 값이다.
OPEN 지정가 주문이 아직 전부 체결되지 않아 대기 중이다. 새 호가가 올 때 재평가 대상이다.
FILLED 체결 완료 상태다.
CANCELED 사용자가 대기 주문을 취소한 종료 상태다.
REJECTED 체결 불가로 거부된 종료 상태다. 예: 시장가인데 체결 가능한 호가가 없음.

시장가와 지정가의 차이

시장가 주문

지금 체결 가능한 호가를 즉시 먹는다. 한 건이라도 체결되면 FILLED로 종료한다. 체결 가능한 호가가 없으면 REJECTED다. MVP에서는 보이는 호가 레벨 안에서 가능한 만큼만 체결하고, 남은 시장가 잔량을 OPEN 대기 주문으로 보관하지 않는다.

지정가 주문

매수 지정가는 지정가 이하의 매도 호가에만 체결되고, 매도 지정가는 지정가 이상의 매수 호가에만 체결된다. 전량 체결되지 않으면 남은 수량은 OPEN으로 남아 이후 호가에서 다시 평가된다.

주문은 사용자의 의도이고, 체결은 그 의도가 시장 가격에 닿은 결과다.

체결 도메인

체결은 주문 1건이 하나 이상의 호가 레벨을 먹으면서 생긴 결과다. 한 주문은 가격이 다른 여러 체결로 쪼개질 수 있다.

호가 레벨 소진 규칙

주문 방향 먹는 호가 가격 순서
BUY asks 가장 싼 매도 호가부터 비싼 쪽으로
SELL bids 가장 비싼 매수 호가부터 싼 쪽으로
호가 레벨 소진 수량 규칙 이번 레벨 체결 수량 = min(남은 주문 수량, 해당 호가 레벨 수량) 내 주문의 남은 수량이 더 작으면 남은 수량까지만 체결되고, 호가 레벨의 잔량이 더 작으면 그 호가에 있는 수량까지만 체결된다. 남은 주문 수량 = 남은 주문 수량 - 이번 레벨 체결 수량
규칙 1. 체결가는 먹은 호가 레벨의 가격이다.

주문 가격이 아니라 실제 닿은 호가 레벨의 가격이 체결가다. 그래서 여러 레벨을 먹으면 체결가가 다른 여러 체결이 생긴다.

규칙 2. 지정가는 한도를 넘는 순간 멈춘다.

BUY 지정가는 limit price보다 비싼 ask를 먹지 않는다. SELL 지정가는 limit price보다 싼 bid를 먹지 않는다. 가격순으로 훑기 때문에 한도를 벗어난 첫 레벨이 나오면 그 뒤도 볼 필요가 없다.

규칙 3. 포지션과 손익은 체결 기록으로만 바뀐다.

주문은 사용자의 의도이고, 체결은 실제로 거래된 결과다. 따라서 포지션과 손익은 주문 상태가 아니라 체결 목록을 기준으로 계산한다. OPEN 주문은 아직 체결되지 않았으므로 포지션과 손익을 바꾸지 않는다.

평균 체결가

평균 체결가 = Σ(체결가 × 체결수량) / Σ체결수량

평균 체결가는 단순 평균이 아니라 수량가중평균이다. 100에 1개, 200에 9개 체결된 주문의 평균 체결가는 150이 아니라 190이다.

체결 도메인의 핵심은 “의도한 주문”을 “실제로 거래된 가격/수량 기록”으로 바꾸는 것이다.

대기 주문 재평가 도메인

지정가 주문이 OPEN으로 남으면 사용자의 추가 요청이 없어도 새 호가가 들어올 때마다 체결 가능성을 다시 평가해야 한다.

실제 거래소에서는 OPEN 지정가 주문이 order book에 올라가고, 반대편 주문 이벤트가 들어와 가격이 맞을 때 매칭 엔진이 가격/시간 우선순위에 따라 체결한다.

하지만 이 프로젝트는 자체 order book을 운영하지 않으므로, 새 Binance 호가 스냅샷이 들어올 때마다 DB의 OPEN 주문을 현재 호가에 다시 대보는 방식으로 단순화한다.

규칙 1. OPEN 주문만 재평가한다.

FILLED, CANCELED, REJECTED는 종료 상태다. 새 호가가 와도 다시 체결 대상이 되지 않는다.

규칙 2. 잔량만 재평가한다.

부분 체결된 지정가 주문을 다시 평가할 때는 원 주문 수량이 아니라 아직 체결되지 않은 잔량만 본다. 이미 체결된 수량을 다시 체결하면 과체결이 된다.

잔량 = 주문 수량 - 누적 체결 수량
규칙 3. 취소와 체결 경쟁은 OPEN 조건으로 방어한다.

재평가 중 사용자가 주문을 취소할 수 있다. 따라서 체결 결과를 저장하기 직전에 아직 OPEN인지 다시 확인해야 한다. 이미 취소된 주문에는 체결을 저장하지 않는다.

OPEN 주문 수 최적화

OPEN 주문이 하나도 없으면 새 호가가 와도 재평가할 대상이 없다. 따라서 OPEN 주문 개수를 메모리 카운터로 캐싱해두고, 0건이면 DB 조회 없이 재평가를 스킵하는 fast-path로 불필요한 작업을 줄인다.

여기서 캐싱은 매번 DB에 묻지 않도록 OPEN 주문 수를 메모리에 보관한다는 뜻이고, fast-path는 결과가 뻔한 경우를 빠르게 건너뛰는 짧은 처리 경로라는 뜻이다.

대기 주문 재평가는 사용자의 요청이 아니라 시장 데이터 변화가 일으키는 내부 업무 흐름이다.

포지션/PnL 도메인

포지션은 주문이 아니라 체결로 바뀐다. 현재 구현에서는 현재 사용자의 모든 paper_orders를 기준으로 그 주문들에 연결된 체결 기록을 찾고, 해당 사용자의 모든 paper_fills를 id 오름차순으로 재생한다. 이 전체 체결 순서대로 계산하면 현재 남은 수량, 평균 진입가, 누적 실현손익을 얻을 수 있다.

포지션 상태

상태 수량 부호 의미
롱 양수 가격 상승에 베팅한 열린 포지션
숏 음수 가격 하락에 베팅한 열린 포지션
flat 0 열린 포지션 없음

상태 전이(체결로 포지션 상태가 바뀌는 과정) 규칙

상황 규칙
포지션 없음 + 체결 체결 방향으로 새 포지션을 연다. 평균 진입가는 단일 체결이면 체결가, 복수 체결이면 수량가중평균 체결가다.
같은 방향 체결 수량을 늘리고, 기존 포지션의 진입 금액과 추가 체결 금액을 합산해 평균 진입가를 다시 계산한다.
반대 방향 체결, 수량 작음 일부 청산한다. 청산 수량에서 실현손익이 발생하고, 남은 포지션의 평균 진입가는 유지한다.
반대 방향 체결, 수량 같음 전부 청산한다. 전체 수량에서 실현손익이 발생하고, 포지션 상태는 flat이 된다.
반대 방향 체결, 수량 큼 전부 청산 후 초과 수량으로 반대 포지션을 새로 연다. 새 평균 진입가는 초과 수량의 체결가 기준으로 계산한다.

계산식

평균 진입가 = sum(체결 수량 × 체결가) / sum(체결 수량) 추가 평균 체결가 = sum(추가 체결 수량 × 추가 체결가) / sum(추가 체결 수량) 같은 방향 체결 후 새 평균 진입가 = (기존 수량 × 기존 평균 진입가 + 추가 수량 × 추가 평균 체결가) / (기존 수량 + 추가 수량) 닫히는 수량 = 기존 포지션에서 실제로 청산되어 없어지는 수량 = min(이번 반대 체결 수량, 기존 포지션 수량) 롱 실현손익 = (청산 가격 - 평균 진입가) × 닫힌 수량 숏 실현손익 = (평균 진입가 - 청산 가격) × 닫힌 수량 미실현손익 = (현재 가격 - 평균 진입가) × 부호 있는 현재 수량

실현손익은 이미 닫힌 부분의 확정 손익이다. 미실현손익은 아직 열린 포지션을 현재 가격으로 평가한 값이라 현재 가격이 없으면 계산할 수 없다.

포지션 도메인의 기준은 체결 원장이다. 주문 자체는 포지션을 바꾸지 않는다.

레버리지/마진 도메인

이 프로젝트의 MVP는 격리 마진, 유지증거금률 0, 수수료 0을 가정한다. 마진은 포지션을 잡기 위해 잠기는 담보금(증거금)이다.

레버리지 두 종류

구분 의미
계좌 레버리지 사용자가 버튼으로 바꾸는 현재 설정값이다. 신규 주문의 매수가능 검증과 주문 기록에 쓰인다.
포지션 레버리지 이미 열린 포지션에 고정된 레버리지다. 포지션이 처음 열리거나 방향이 뒤집혀 새 포지션이 생긴 시점의 계좌 레버리지를 사용한다. 사용 증거금과 청산가는 이 값으로 계산한다.

계좌 레버리지는 다음 주문에 적용되는 설정값이다. 이미 열린 포지션의 사용 증거금과 청산가가 계좌 레버리지 변경으로 흔들리면 안 되므로, 격리 마진(증거금)에서는 포지션 레버리지를 별도로 들고 간다.

마진 계산식(여기서 마진은 포지션 담보금/증거금)

거래량 가중 평균 가격(Volume Weighted Average Price, VWAP) 평균 진입가(VWAP) = sum(체결 수량 × 체결가) / sum(체결 수량) 진입 명목금액 = sum(진입 체결 수량 × 체결가) = 평균 진입가 × |포지션 수량| 사용 증거금 = 진입 명목금액 / 포지션 레버리지 롱 청산가 = 평균 진입가 - 평균 진입가 / 포지션 레버리지 숏 청산가 = 평균 진입가 + 평균 진입가 / 포지션 레버리지 MVP 단순 청산가: 유지증거금률, 수수료, 펀딩비를 0으로 두고 손실이 사용 증거금을 모두 소진하는 가격으로 계산한다. 실제 거래소의 청산가는 유지증거금률, 명목금액 tier(포지션 규모별 구간), 마진 모드, 추가 마진 등에 따라 달라진다. 유지증거금률은 포지션을 계속 유지하기 위해 남겨둬야 하는 최소 담보 비율이다. 거래소는 손실이 담보금을 완전히 소진한 뒤 청산하면 늦을 수 있으므로, 최소 안전선 아래로 내려가기 전에 청산한다.

신규 주문 매수가능 검증

새로 열거나 같은 방향으로 늘리는 수량은 추가 증거금이 필요하다. 반대 방향 주문이 기존 포지션을 줄이거나 닫는 순수 청산 주문이면 새 증거금이 필요 없다.

신규/같은 방향 주문의 여는 수량 = 주문 수량 전부 반대 방향 주문의 여는 수량 = max(주문 수량 - |기존 포지션 수량|, 0) 필요 증거금 = 여는 수량 × 기준 가격 / 계좌 레버리지

시장가 주문의 기준 가격은 현재 mid, 지정가 주문의 기준 가격은 limit price다.

마진 도메인의 핵심은 “새로 위험을 늘리는 수량”에만 추가 담보금이 필요하다는 점이다.

계좌 평가 도메인

계좌 테이블에는 시드 현금과 현재 계좌 레버리지만 저장한다. 지갑잔고, 평가자산, 사용 증거금, 가용잔고는 저장 컬럼이 아니라 화면 응답을 만들 때 계산하는 평가값이다. 이 값들은 사용자 계좌, 체결 원장, 주문별 레버리지, 현재 호가를 합쳐 계산한다.

MVP 계좌 평가 계산식

아래 계산식은 실제 바이낸스 계좌 규칙 전체가 아니라, 현재 프로젝트의 단일 심볼, 격리 마진, 수수료/펀딩비 0 가정에 맞춘 화면 평가식이다.

지갑잔고 = 시드 현금 + 누적 실현손익 평가자산 = 지갑잔고 + 미실현손익 사용 증거금 = 진입 명목금액 / 포지션 레버리지 가용잔고 = max(지갑잔고 - 사용 증거금, 0)

포지션 표시 규칙

부호 있는 수량 양수 side = LONG, quantity = 절대값
부호 있는 수량 음수 side = SHORT, quantity = 절대값
부호 있는 수량 0 position = null

청산(Liquidation) 도메인

청산(liquidation)은 손실이 커진 포지션을 시스템이 강제로 닫는 규칙이다. mark는 손익과 청산 판정에 쓰는 기준 가격이다. 실제 거래소는 순간적으로 튄 마지막 체결가만으로 청산하지 않도록 별도 mark price를 쓰지만, 이 프로젝트는 현재 호가의 중간값인 mark(mid) = (best bid + best ask) / 2를 기준 가격으로 쓴다. 현재 mark(mid)가 포지션의 청산가에 닿으면 청산 대상이 된다. MVP에서는 유지증거금률, 수수료, 펀딩비를 0으로 보므로, 손실이 사용 증거금을 모두 소진하는 가격을 청산가로 둔다.

청산가 계산

청산은 손실이 사용 증거금과 같아지는 지점이다. 먼저 현재가를 X로 놓고, 청산 조건을 만족하는 X를 마지막에 청산가로 부른다.

E = 진입가, q = 수량, L = position leverage, X = 현재가(mark) 명목금액 = E × q 사용 증거금 = (E × q) / L 롱 손실 = (E - X) × q 청산 조건: (E - X) × q = (E × q) / L 양쪽을 q로 나누면: E - X = E / L 정리: X = E - E / L 따라서 롱 청산가 = E - E / L = 진입가 - 진입가 / L 숏 청산가 = E + E / L = 진입가 + 진입가 / L
예시: 진입가 50,000, L = 10 -> 진입가 / L = 5,000 -> 롱 청산가 = 45,000

청산 판정

포지션 청산 조건
롱 현재 mark(mid)가 청산가 이하이면 청산 대상이다.
숏 현재 mark(mid)가 청산가 이상이면 청산 대상이다.
롱: mark <= liquidationPrice 숏: mark >= liquidationPrice

강제 청산 주문/체결 기록 규칙

실제 거래소의 강제청산은 포지션 수량만큼 반대 방향으로 닫는 행위라, 유동성 상황에 따라 여러 가격 레벨에서 체결되어 fill이 여러 건 생길 수 있다. 이 프로젝트의 MVP에서는 청산이 발생하면 반대 방향 MARKET 주문 1건과 청산가 fill 1건을 기록한다. 롱은 SELL로 닫고, 숏은 BUY로 닫는다. 실제 거래소처럼 호가 레벨별로 여러 체결을 만들지는 않는다.

기존 포지션 강제 청산 방향
롱 SELL
숏 BUY

청산 감시 규칙

청산 감시는 주문 제출 여부와 무관하게, 열린 포지션이 가격 변화 때문에 유지 조건을 벗어나는지 계속 평가하는 규칙이다. 이 프로젝트의 MVP에서는 1초마다 최신 mark(mid)가 단순 청산가에 닿았는지만 확인한다. 실제 Binance Futures의 mark price는 단순 호가 mid가 아니라, index price와 premium/funding 정보를 바탕으로 거래소가 산출한 가격이다.

청산 도메인은 손실이 사용 증거금을 모두 소진하는 지점에서 시스템이 포지션을 강제로 닫는 MVP 안전 규칙이다.

현재 MVP 가정

  • 거래 심볼은 BTCUSDT 하나만 지원한다.
  • 한 심볼은 롱/숏을 동시에 들지 않는 순포지션 모드로 관리한다.
  • 수수료와 펀딩비는 0으로 둔다.
  • 마진은 격리 마진으로 보고, 유지증거금률은 0으로 둔다.
  • 청산가는 단순 모델로 계산한다.
  • 체결 후 실제 거래소 호가 수량을 차감하지 않는다.
  • 시장가 잔량은 OPEN으로 남기지 않는다.
  • 체결 기록은 수정하거나 삭제하지 않고 계속 쌓아 둔다. 현재 포지션, 평균 진입가, 실현손익은 필요할 때 DB에 저장된 체결 기록을 첫 체결부터 순서대로 다시 읽어 계산한다.
