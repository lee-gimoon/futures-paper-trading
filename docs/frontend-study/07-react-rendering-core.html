React가 화면을 바꾸는 핵심

컴포넌트 함수는 화면을 계산하고, React는 상태를 기억하며, ReactDOM은 변경된 DOM만 반영한다.

UI = f(props, state)

1 왜 React를 쓰는가?

직접 DOM 조작 요소 찾기
→ 값 변경
→ 경우마다 동기화
→
React 선언형 UI 상태 변경
→ 원하는 화면 계산
→ 필요한 DOM만 반영

2 “React 엔진”의 정확한 역할

Browser

JS 엔진

JavaScript 실행

react

React

상태·컴포넌트 트리·갱신 관리

react-dom

ReactDOM

React 결과를 DOM에 연결

Browser

DOM·렌더러

실제 화면 표시

정확히는: React는 브라우저/Node의 JS 엔진이 아니라, node_modules에서 가져와 브라우저가 실행하는 UI 라이브러리다.

3 함수형 컴포넌트

function LoginForm(props) {
  const [email, setEmail] = useState('');
  return <input value={email} />;
}
LoginForm 함수
현재 props·state로 JSX 계산
⇄
React 내부 관리 정보
트리의 사용 위치별 상태 기억

new LoginForm() 객체는 없다. 함수는 계산하고, React가 각 <LoginForm /> 사용 위치를 구분한다.

4 Hook: useState

const [email, setEmail] = useState('');
// 입력: 첫 상태의 초깃값 ''
// 반환: [현재 렌더링의 상태 값, 변경 요청 함수]
매 렌더링 호출

useState도 함수와 함께 다시 호출된다.

초깃값은 처음만

다음 렌더링에는 React가 기억한 값이 반환된다.

setter는 예약

setEmail은 값을 즉시 바꾸지 않고 갱신을 요청한다.

5 상태가 바뀌면 화면이 바뀌는 한 사이클

사용자 입력

onChange 이벤트 발생

setter 호출

setEmail('a')

React가 기록

업데이트를 저장하고 렌더링 예약

함수 재실행

LoginForm()이 새 JSX 계산

차이 비교

이전 JSX 결과와 새 결과 비교

DOM 반영

ReactDOM이 바뀐 부분만 수정

다시 렌더링 ≠ DOM 전체 재생성. 컴포넌트 함수는 다시 계산되지만 실제 DOM에는 필요한 변경만 적용된다.

6 어떤 컴포넌트가 다시 계산되는가?

App의 setForm→App에서 로그인 폼 포함 여부 재계산
LoginForm의 setEmail→LoginForm의 입력 화면 재계산

7 시작 코드는 어디인가?

ReactDOM.createRoot(document.getElementById('root'))
  .render(<App />);

main.tsx가 React 관리 영역을 만들고 최초 렌더링을 시작한다. 이후에는 이벤트와 setter가 다음 렌더링을 요청한다.

함수는 계산한다 · React는 기억한다 · ReactDOM은 DOM에 반영한다
